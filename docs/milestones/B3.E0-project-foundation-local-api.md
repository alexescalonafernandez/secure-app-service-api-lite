# B3.E0 - Project foundation + local Minimal API

## Objective
Establish a verifiable local foundation for Project B3 with a working Minimal API, request validation, queue abstraction, and baseline automated tests.

## What was implemented
- ASP.NET Core Minimal API on .NET 8.
- `GET /health` endpoint with simple health payload.
- `POST /api/messages` endpoint accepting message submission.
- Request/response contracts for message creation.
- `IMessageQueue` abstraction for enqueue behavior.
- `InMemoryMessageQueue` fake implementation for local-only queueing.
- FluentValidation rules for request correctness.
- Swagger/OpenAPI in Development.
- Initial endpoint and validator test coverage.

## Endpoints
- `GET /health`
- `POST /api/messages`

## Validation rules
- `subject` is required and limited to 120 chars.
- `body` is required and limited to 2000 chars.
- `priority` is optional; allowed values: `Low`, `Normal`, `High` (case-insensitive).
- Missing/blank `priority` is normalized to `Normal` when queued.

## Tests added
- Health endpoint response contract test.
- Message endpoint accepted response test for valid payload.
- Message endpoint bad request tests for invalid subject/body/priority.
- Validator unit tests for success and failure conditions.

## Manual validation performed
- Launched API locally.
- Called `/health` and verified `200 OK` payload.
- Called `/api/messages` with valid payload and verified `202 Accepted`.
- Sent invalid payloads and verified validation problem responses.
- Opened Swagger UI in Development and validated endpoint metadata visibility.

## Lessons learned from using Copilot/Codex
- Best results came from explicit, bounded prompts with acceptance criteria.
- Validation rules and test expectations should be stated in prompt text, not inferred.
- Iterative review is necessary to keep generated output aligned with milestone boundaries.
- Issues are useful for traceability, but AI implementation prompts should be self-contained, explicit, and verifiable instead of relying on the model to read a specific issue comment.
