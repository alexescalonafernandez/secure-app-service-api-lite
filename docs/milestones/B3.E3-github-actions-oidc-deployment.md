# B3.E3 - GitHub Actions deployment with OIDC

## Objective
B3.E3 introduced secure GitHub Actions deployment to Azure App Service using OpenID Connect (OIDC) and performed the first real end-to-end Azure validation of `POST /api/messages` reaching Azure Storage Queue.

## What was implemented
- OIDC bootstrap PowerShell script.
- OIDC validation PowerShell script.
- GitHub Actions OIDC setup documentation.
- Build/test GitHub Actions workflow.
- App Service deployment workflow.
- Azure login with OIDC.
- Manual deployment workflow using `workflow_dispatch`.
- End-to-end Azure validation.

## Deployment identity model
- GitHub Actions uses OIDC to authenticate to Azure.
- No Azure client secret is stored in GitHub.
- No App Service publish profile is used.
- The GitHub-issued OIDC token is trusted through a Microsoft Entra App Registration, Service Principal, and federated credential.
- The deployment identity receives Contributor access at the Web App resource scope only.
- The federated credential subject is:

```text
repo:alexescalonafernandez/secure-app-service-api-lite:environment:dev
```

## Runtime identity model
- Azure App Service uses a system-assigned Managed Identity at runtime.
- The runtime identity accesses Azure Storage Queue.
- The required RBAC role is Storage Queue Data Contributor from B3.E2.
- The runtime identity and deployment identity are separate identities with separate responsibilities.

## GitHub Environment setup
The GitHub Environment `dev` is required for the deployment workflow.

Environment secrets:
- `AZURE_CLIENT_ID`
- `AZURE_TENANT_ID`
- `AZURE_SUBSCRIPTION_ID`

Environment variables:
- `AZURE_WEBAPP_NAME = app-b3-secure-api-dev-we-01`
- `DOTNET_VERSION = 8.0.x`
- `PROJECT_PATH = src/SecureAppServiceApiLite.Api/SecureAppServiceApiLite.Api.csproj`

## Workflows added
- `.github/workflows/build-test.yml`
- `.github/workflows/deploy-webapp.yml`

## Deployment workflow behavior
The deployment workflow:
- runs manually using `workflow_dispatch`;
- checks out code;
- sets up .NET;
- restores dependencies;
- builds the full solution;
- runs tests with `--no-build`;
- publishes only the API project using `PROJECT_PATH`;
- authenticates to Azure using `azure/login` and OIDC;
- deploys to Azure App Service using `azure/webapps-deploy`.

## End-to-end validation performed
- Deploy Web App workflow completed successfully.
- `GET /health` worked in Azure.
- `POST /api/messages` worked in Azure.
- The message was confirmed in Azure Portal in the Storage Queue `incoming-messages`.
- Runtime path validated:

```text
POST /api/messages in Azure
  -> AzureStorageMessageQueue
  -> DefaultAzureCredential
  -> App Service Managed Identity
  -> Storage Queue Data Contributor
  -> Azure Storage Queue
```

## Troubleshooting and lessons learned
- The initial deployment workflow failed because the build step targeted only the API project while the test step used `dotnet test --no-build`.
- Because the test project had not been built, `dotnet test --configuration Release --no-build` failed.
- The fix was to build the full solution with `dotnet build --configuration Release --no-restore`.
- `dotnet publish` should remain scoped to `${{ vars.PROJECT_PATH }}` so only the deployable API project is published.
- `az storage message peek --auth-mode login` can fail if the local Azure CLI user lacks data-plane permissions.
- A local CLI data-plane permission failure does not mean the App Service Managed Identity failed.

## Security notes
- OIDC avoids long-lived Azure credentials.
- No publish profile is used.
- No client secret is stored in GitHub.
- Deployment RBAC is scoped to the Web App resource.
- Runtime identity and deployment identity remain separate.
- Do not commit script output values.

## Cost-control notes
- App Service B1 can generate cost while deployed.
- The existing teardown script remains the cost-control mechanism.
- The deployment workflow does not create new Azure infrastructure.

## Known limitations
- Deployment is manual via `workflow_dispatch`, not automatic on push.
- No deployment slots.
- No rollback strategy.
- No production networking.
- No Key Vault.
- No queue processor/consumer.
- Azure Portal was used for queue message confirmation because local CLI data-plane access failed.

## Out of scope
- Key Vault.
- Production networking.
- Private endpoints.
- Custom domains.
- Deployment slots.
- Blue/green deployment.
- Containerization.
- Kubernetes.
- Queue processor/consumer worker.
