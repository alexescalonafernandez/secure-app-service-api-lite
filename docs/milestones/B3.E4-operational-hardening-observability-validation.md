# B3.E4 - Operational hardening and observability validation

## Objective
B3.E4 focused on proving that the deployed API can be operated and troubleshot at a basic level with the existing Azure App Service, Application Insights, Storage Queue, and GitHub Actions deployment path. This milestone did not add new business features.

## What was implemented
- Created the initial App Service operations runbook.
- Integrated the Application Insights SDK into the API.
- Registered Application Insights conditionally only when an Application Insights connection string is available.
- Validated successful and invalid API requests in Azure.
- Documented queue verification, Log Stream behavior, identity caveats, and the troubleshooting flow.

## Operational evidence model
Application Insights is the primary operational evidence source for the current project scope. It is used to confirm request success, request failure, result code, and request duration for the deployed API.

## Application Insights enablement
Application Insights telemetry requires all of these pieces to be present:

- `APPLICATIONINSIGHTS_CONNECTION_STRING` configured in Azure App Service.
- `Microsoft.ApplicationInsights.AspNetCore` referenced by the API project.
- `builder.Services.AddApplicationInsightsTelemetry()` registered only when a connection string is available.

This keeps Azure telemetry enabled for the deployed app while allowing local execution and tests to run without requiring a real Application Insights connection string.

## Validation performed
- The deployment workflow completed successfully before validation.
- `GET /health` returned HTTP `200` in Azure and appeared in Application Insights.
- A valid `POST /api/messages` returned HTTP `202` in Azure and appeared in Application Insights.
- An invalid `POST /api/messages` returned HTTP `400` in Azure and appeared in Application Insights.
- A queue message was confirmed in the Azure Portal under the `incoming-messages` queue in storage account `stb3sapidevwe01`.

## App Service Log Stream findings
- Runtime Log Stream connected successfully.
- Platform Log Stream connected successfully.
- No normal request logs appeared during the reviewed requests.
- This is expected because the API currently has no explicit `ILogger` application events and App Service file-system logging is not enabled.
- This finding is not an error for the current milestone scope.

## Identity and permission considerations
The project uses separate identities for separate responsibilities:

- Local Azure CLI identity: used for manual `az` commands from a developer workstation or shell.
- GitHub Actions OIDC identity: used by the deployment workflow.
- App Service system-assigned Managed Identity: used by the running API for queue access.

A local Azure CLI permission failure does not prove that the App Service runtime Managed Identity cannot access the queue. Runtime queue access must be evaluated from the deployed application path and its assigned Azure RBAC permissions.

## Operations runbook
Operational procedures, evidence checks, identity caveats, troubleshooting flow, and current limitations are documented in `docs/operations/app-service-runbook.md`.

## Troubleshooting lessons learned
- An Application Insights resource and App Service connection string are insufficient without application SDK registration.
- Conditional telemetry registration keeps local execution and tests independent from a real Application Insights connection string.
- Application Insights is more useful than Log Stream for the current API scope.
- Build the full solution before using `dotnet test --no-build` in CI, while publish should target only the deployable API project.

## Security notes
- OIDC is used for deployment.
- No Azure client secret is stored in GitHub.
- No App Service publish profile is used.
- Deployment identity and runtime identity are separate.
- Future structured logging must not log message bodies or sensitive data.

## Cost-control notes
- App Service B1 incurs cost while deployed.
- The deployment workflow does not create infrastructure.
- Existing teardown tooling remains the cost-control mechanism.

## Known limitations
- Deployment remains manual through `workflow_dispatch`.
- No alerts or dashboards are configured.
- No explicit structured application logging exists.
- No queue consumer exists.
- No retry policy, poison-message handling, or dead-letter handling exists.
- No deployment slots are configured.
- No production networking hardening is configured.
- Local Azure CLI queue inspection may require Storage Queue data-plane permissions.

## Out of scope

- Queue consumer/worker.
- Retry and dead-letter handling.
- Deployment slots.
