# B3.E2 - Managed Identity + RBAC + real Azure Storage Queue integration path

## Objective
B3.E2 connects the App Service system-assigned Managed Identity and the Azure Storage Queue integration path while keeping local development safe and simple with the in-memory provider by default.

## What was implemented
- RBAC Bicep module.
- Storage Queue Data Contributor role assignment for the App Service Managed Identity.
- Azure Storage Queue implementation behind `IMessageQueue`.
- Queue provider configuration model.
- Provider registration extension.
- App Service provider setting for Azure.
- Provider registration tests.
- Queue provider integration documentation in `docs/integration/queue-provider.md`.

## Infrastructure changes
- `infra/modules/rbac.bicep` was added.
- `infra/main.bicep` wires the RBAC module into the deployment.
- `infra/modules/app-service.bicep` sets `QueueOptions__Provider = AzureStorage`.
- The role assignment uses the built-in Storage Queue Data Contributor role.
- The role assignment is scoped to the Storage Account.

## Application changes
- `AzureStorageMessageQueue` was added.
- `QueueOptions` was added.
- `MessagingServiceCollectionExtensions.AddMessageQueue(...)` was added.
- `Program.cs` now delegates queue registration to the extension.
- The `Azure.Identity` and `Azure.Storage.Queues` packages were added.
- `appsettings.json` defines `InMemory` as the default provider.

## Configuration model
The queue provider uses the following configuration keys:
- `QueueOptions:Provider`
- `QueueOptions:StorageAccountName`
- `QueueOptions:QueueName`

Azure App Service uses the equivalent double-underscore app setting names:
- `QueueOptions__Provider`
- `QueueOptions__StorageAccountName`
- `QueueOptions__QueueName`

## Local vs Azure behavior
- Local/default execution uses `InMemoryMessageQueue`.
- Azure App Service uses `AzureStorageMessageQueue` because Bicep sets `QueueOptions__Provider = AzureStorage`.
- Local development does not require Azure credentials by default.
- Azure relies on Managed Identity and RBAC.

## Managed Identity and RBAC
- App Service has a system-assigned Managed Identity.
- Bicep assigns Storage Queue Data Contributor to that identity.
- The assignment is scoped to the Storage Account.
- RBAC propagation may take time after deployment.

## Tests added
Provider registration tests verify that:
- A missing/default provider resolves `InMemoryMessageQueue`.
- Provider `InMemory` resolves `InMemoryMessageQueue`.
- Provider `AzureStorage` resolves `AzureStorageMessageQueue`.
- Provider `azurestorage` resolves `AzureStorageMessageQueue`.
- The tests do not call Azure or send real queue messages.

## Validation performed
- `dotnet build` was executed successfully.
- `dotnet test` was executed successfully.
- The app was run locally and `POST /api/messages` was manually tested using the default in-memory provider.
- The Bicep scripts were used during the RBAC and app settings changes as applicable: build, validate, and what-if.

## What was not validated yet
- No live deployed application enqueue to Azure Storage Queue was validated yet.
- Full end-to-end Azure validation requires application deployment.
- Application deployment is intentionally deferred to B3.E3.

## Security notes
- No Storage connection strings are used.
- Identity-based access is preferred.
- Key Vault is still deferred because there is no real secret yet.
- `DefaultAzureCredential` uses Managed Identity in Azure App Service.

## Cost-control notes
- B3.E2 does not add new long-running resources beyond those from B3.E1.
- The existing B1 App Service Plan can generate cost if deployed and not torn down.
- The existing teardown script from B3.E1 remains the cost-control mechanism.

## Known limitations
- No GitHub Actions deployment yet.
- No end-to-end Azure queue enqueue validation yet.
- No retry/backoff policy yet.
- No poison message handling yet.
- No queue processor/consumer yet.
- RBAC propagation delay can affect first runtime tests.
