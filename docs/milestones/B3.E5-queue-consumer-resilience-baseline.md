# B3.E5 - Queue consumer and resilience baseline

## Status

Completed.

## Objective

B3.E5 extended the project from a simple API queue producer into an Azure-based asynchronous processing flow. The milestone proved that messages accepted by the App Service API can be enqueued into Azure Storage Queue, consumed by a serverless Azure Function, observed through Application Insights, retried on failure, and moved to a poison queue after retry exhaustion.

The objective was not production hardening. It was to establish a validated resilience baseline for the queue consumer path, including operational telemetry, sanitized logging, retry behavior, poison-message handling, and practical KQL troubleshooting evidence.

## Scope delivered

- Shared queue message contract.
- API producer using Azure Storage Queue.
- Azure Function queue consumer using .NET 8 isolated worker.
- Sanitized structured telemetry for queue processing outcomes.
- Function App infrastructure on Azure Functions Flex Consumption.
- Function runtime Managed Identity.
- Identity-based queue trigger connection.
- Manual GitHub Actions deployment workflow using OIDC.
- Azure happy-path validation.
- Azure retry and poison-message validation.
- Application Insights KQL troubleshooting guide.

## Architecture

Validated Azure path:

```text
Client/Postman
  -> API App Service
  -> Azure Storage Queue incoming-messages
  -> Azure Function IncomingMessagesQueueTrigger
  -> Application Insights
```

Messages that fail processing until retry exhaustion move to the poison queue:

```text
incoming-messages-poison
```

## Identity and RBAC

B3.E5 kept deployment and runtime responsibilities separated by identity:

- API runtime Managed Identity enqueues messages.
- Function runtime Managed Identity consumes messages.
- GitHub Actions OIDC identity deploys code.

The Function runtime Managed Identity uses the following Azure Storage Queue data-plane roles on the source storage account:

- `Storage Queue Data Reader`
- `Storage Queue Data Message Processor`
- `Storage Queue Data Message Sender`

`Storage Queue Data Reader` and `Storage Queue Data Message Processor` were sufficient for normal queue trigger processing, but they were not sufficient for poison queue handling. After retry exhaustion, the Azure Functions runtime must also enqueue the failed message into `incoming-messages-poison`; that operation requires sender permission. The milestone fixed this by granting `Storage Queue Data Message Sender` instead of broadening access to `Storage Queue Data Contributor`.

## Validation evidence

### Happy path

The deployed API health endpoint returned a healthy response:

```json
{
  "status": "Healthy",
  "service": "SecureAppServiceApiLite.Api"
}
```

Posting a valid message to the API returned `202 Accepted` with the accepted message identifier:

```json
{
  "messageId": "db68da0e-af6e-463f-b8b4-e20ef8516c5c",
  "status": "Accepted"
}
```

Validation confirmed that:

- `incoming-messages` became empty after Function processing.
- Application Insights `requests` contained `IncomingMessagesQueueTrigger` with `success == true`.
- Application Insights `traces` contained the successful processing event:

```text
Queue message processed. MessageId: db68da0e-af6e-463f-b8b4-e20ef8516c5c; Priority: Normal; Outcome: Succeeded
```

- The processed `MessageId` matched the API response.
- Sanitization checks showed no `Body` or payload content in telemetry.

### Retry and poison path

An invalid message was inserted directly into `incoming-messages` to validate rejection, retry, and poison queue behavior:

```json
{
  "Id": "22222222-2222-2222-2222-222222222222",
  "Subject": "Azure poison validation",
  "Body": "this-body-must-not-appear-in-logs",
  "Priority": "Urgent",
  "CreatedAtUtc": "2026-07-17T15:45:00Z"
}
```

Validation confirmed that:

- The Function rejected the message because the envelope was invalid.
- Application Insights `traces` contained the rejected processing event:

```text
Queue message rejected. FailureReason: InvalidEnvelope; Outcome: Rejected
```

- Application Insights `requests` showed failed `IncomingMessagesQueueTrigger` invocations.
- After retries, the invalid message moved to `incoming-messages-poison`.
- Sanitization checks showed no `Body`, payload, or `this-body-must-not-appear-in-logs` value in telemetry.

### RBAC troubleshooting

During Azure retry and poison queue validation, the Function runtime initially failed to copy the failed message to the poison queue:

```text
Azure.RequestFailedException
Status: 403
ErrorCode: AuthorizationPermissionMismatch
QueueProcessor.CopyMessageToPoisonQueueAsync
QueueClient.SendMessageAsync
```

The diagnosis was that normal queue processing permissions did not include permission to send the failed message to the poison queue. The fix was to explicitly provision `incoming-messages-poison`, keep the existing reader and message processor roles, and add `Storage Queue Data Message Sender` for the Function runtime Managed Identity.

## Risks and limitations

- Deployment remains manual.
- No alert rules or dashboards yet.
- Retry behavior uses Azure Functions runtime defaults.
- No automated poison-message reprocessing workflow.
- No custom dead-letter dashboard.
- No idempotency store.
- No private networking.
- No Key Vault.
- No production hardening.
