# B3.E1 - Azure infrastructure skeleton with Bicep

## Objective
B3.E1 establishes the first Azure infrastructure skeleton for Project B3 using Bicep, focused on provisioning platform resources only. This milestone does **not** deploy application code yet.

## What was implemented
- Bicep root file (`infra/main.bicep`).
- Environment parameter file using `.bicepparam` (`infra/dev.bicepparam`).
- Storage module (`infra/modules/storage.bicep`).
- Observability module (`infra/modules/observability.bicep`).
- App Service module (`infra/modules/app-service.bicep`).
- PowerShell-oriented `.azcli` scripts for repeatable Azure CLI operations.

## Azure resources provisioned
- Storage Account.
- Storage Queue `incoming-messages`.
- Log Analytics Workspace.
- Workspace-based Application Insights.
- Linux App Service Plan (B1).
- Linux Web App.
- System-assigned Managed Identity.

## Bicep structure
```text
infra/
  main.bicep
  dev.bicepparam
  modules/
    storage.bicep
    observability.bicep
    app-service.bicep
  scripts/
    00-account-context.azcli
    01-build.azcli
    02-validate.azcli
    03-what-if.azcli
    04-deploy.azcli
    05-inspect.azcli
    99-teardown.azcli
```

## Parameterization
- `location = westeurope`
- `environment = dev`
- `projectCode = b3sapi`
- `locationCode = we`
- `instance = 01`
- `.bicepparam` is used instead of JSON parameter files.

## Azure CLI scripts
The CLI scripts are intended to be executed from VS Code on Windows using Azure CLI from PowerShell. For that reason, multiline command continuation uses PowerShell backticks (`` ` ``) rather than Bash backslashes (`\`).

## Validation performed
The following scripts were executed successfully:
- `01-build.azcli`
- `02-validate.azcli`
- `03-what-if.azcli`

## Deployment performed
`04-deploy.azcli` was executed successfully and deployed the skeleton infrastructure to:
- `rg-b3-secure-api-dev-we-01`

## Inspection performed
`05-inspect.azcli` was executed successfully and inspected:
- Resource group resources.
- App Service summary.
- App Service app setting names.
- Storage Queue existence.

## Teardown performed
`99-teardown.azcli` was executed successfully and deleted the dev resource group to avoid ongoing cost.

## Cost-control notes
- App Service Plan B1 can generate cost while deployed.
- Log Analytics/Application Insights may also generate cost depending on ingestion.
- The lab flow includes teardown as a required operational step.
- `99-teardown.azcli` deletes the full dev resource group.

## Security notes
- App Service uses a system-assigned Managed Identity.
- Storage Account disables public blob access and enforces HTTPS/TLS 1.2.
- Key Vault is intentionally out of scope because there is no real secret yet.
- RBAC role assignment between the App Service Managed Identity and Storage Queue is intentionally deferred to a later milestone.

## Out of scope
- Application deployment.
- Azure SDK integration in code.
- RBAC role assignments.
- Key Vault.
- GitHub Actions.
- User authentication/authorization.
- Production networking.
- Private endpoints.
- Custom domains.
