# B3.E6 - Operational alerts and dashboard baseline

## Status

Completed.

## Objective

B3.E6 closes Project B3 with a minimal proactive operations baseline. After B3.E5 validated the asynchronous processing path from API to Storage Queue, Azure Function, and Application Insights, this milestone moved the project from manual diagnostics toward a basic ability to detect queue backlog / poison-suspected conditions through Azure Monitor.

## Scope delivered

- ADR-0004 alerting design.
- Azure Monitor Action Group.
- Azure Monitor Metric Alert.
- Queue backlog / poison-suspected alert based on `QueueMessageCount > 0`.
- Hourly evaluation/window because `QueueMessageCount` is an hourly capacity metric.
- Updated `infra/scripts/05-inspect.azcli`.
- Alert validation in Azure Portal.
- Poison-message response runbook.
- Link from Application Insights KQL guide to the runbook.

## Alert design

- Metric: `QueueMessageCount`
- Threshold: `> 0`
- Severity: `3 - Informational`
- Evaluation/window: hourly
- Scope: Storage Queue service level
- Interpretation: queue backlog / poison suspected
- Limitation: not guaranteed poison-queue-specific

`QueueMessageCount` is evaluated at the Storage Queue service level. It is therefore treated as a queue backlog / poison-suspected signal, not as a perfect poison-queue-specific metric.

## Risks and limitations

- Alert is queue-service-level, not poison-queue-specific.
- Hourly evaluation means it is not near-real-time.
- No email receiver is committed by default.
- No automated poison-message reprocessing.
- No production-grade dashboard/workbook.
- No alert routing to Teams/PagerDuty/Slack.
- No Key Vault/private networking.
